事务的ACID

Rrigyouu 2021-01-17 01:20:40

事务的ACID

原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）

原子性：最小分割单元

一致性：一个一致性状态转移到另一个一致性状态

隔离性：最终提交前，对其他事务不可见

持久性：一旦提交，永久保留到数据库中

Mysql事务架构

mysql服务器架构分为三层：

  1. 连接层：客户端连接、授权认证；
  2. 服务器层（缓存模块、语法分析模块、优化模块）
  3. 存储引擎层（只有存储引擎层实现事务）。

事务的隔离性通过锁、MVCC等实现

事务的原子性、一致性、持久性通过事务日志实现

事务隔离级别

SQL定义了四种隔离级别：

隔离级别 要点
read uncommitted 1. 所有事务都可以看到其他未提交事务的执行结果
2. 少用，可能引发脏读
read committed 1. 多数数据库默认隔离级别
2. 事务只能看见已经提交的事务
3. 可能出现不可重复读，即同一事务中完全相同的select数据可能看到不同结果
repeatable read 1. mysql默认事务隔离级别
2. 同一事务的多个实例在并发读取数据时，会看到同样的数据行
3. 可能出现幻读
4. InnoDB通过MVCC和间隙锁解决幻读
serializable 1. 最高隔离级别
2. 强制事务排序，解决幻读问题
3. 可能出现大量超时和锁竞争

并发事务问题

更新丢失：多个事务选择同一行，发生更新后丢失（如同一份文件两个人同时更改）

脏读：事务对修改某条记录，但未提交，此时另一个事务读取同一条数据，读取到了脏数据

不可重复读：再次读取某数据，发现数据已经改变（重点在于数据的修改）

幻读：再次检索某数据，发现其他事务插入了满足检索条件的其他数据（重点在于数据的新增和删除）

并发解决办法

更新丢失：1. 数据库事务控制；2. 应用程序也要加锁。

脏读、不可重复读、幻读：是数据库的一致性问题，通过加锁和多版本并发控制（MultiVersion Concurrency Control，简称 MVCC）实现。

MVCC

原因：并发控制，实现数据库的并发访问。数据库一般读多写少，于是发明了锁（读锁和写锁）；读锁之间不互斥，写锁和所有的锁互斥；但是读锁的并发性能依然不够；提出读写不冲突的方法，读取数据时通过一种类似快照的方式将数据保存下来，这样读锁就和写锁不冲突了，不同的事务session会看到自己特定版本的数据。

实现方式：MVCC的实现是通过保存数据在某一个时间点快照来实现的。

实现原理：InnoDB的MVCC是通过在每行记录后面保存两个隐藏的列来实现。这两个列一个保存了行的创建时间，一个保存行的过期时间（删除时间）。当然存储的并不是真实的时间而是系统版本号（system version number）。每开始一个新的事务，系统版本号都会自动新增。事务开始时刻的系统版本号会作为事务的版本号，用来查询到每行记录的版本号进行比较。

操作 mvcc
select 1. innodb只找早版本遭遇当前事务版本的数据行
2. 行的删除版本号要么未定义，要么大于当前事务版本号，这样可以确保事务读取到的行在事务开始之前未被删除
insert 新插入的行保存当前系统版本号
delete 保存当前系统版本号作为行删除标识
update 保存当前系统版本号作为行版本号，同时保存当前系统版本号到原来的行作为删除标识
