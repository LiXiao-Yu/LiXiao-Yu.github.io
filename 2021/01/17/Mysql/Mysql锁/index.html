Mysql锁

锁是服务层和存储引擎实现并发控制的。

锁的机制

锁机制 说明
共享锁（读锁） 其他事务可读，不可写
排他锁（写锁） 其他事务不能读，也不能写

锁粒度

引擎 锁粒度
MYISAM 表级锁
MEMORY 表级锁
BDB 页面锁，支持表级锁
INNODB 行级锁，支持表级锁
锁粒度 性能
表级锁 自动获得；
开销小，加锁快；不会出现死锁；锁定粒度大，发生锁冲突的概率最高，并发度最低；
行级锁 自动获得；
开销大，加锁慢；会出现死锁；锁定粒度最小，发生锁冲突的概率最低，并发度也最高；
只在InnoDB存储引擎中实现；
页面锁 开销和加锁时间界于表锁和行锁之间；会出现死锁；锁定粒度界于表锁和行锁之间，并发度一般。

MYISAM锁

读锁：不会阻塞同表读请求，阻塞同表读请求

写锁：阻塞同表写请求

读操作和写操作、写操作和写操作串行。

默认写锁比读锁优先级更高，当锁释放时优先给写锁在给读锁，因此大量更新和查询操作不适合用MYISAM引擎。

INNODB锁

只有通过索引，INNODB才会使用行锁；否则，使用表锁。

行锁是针对索引加的锁，而不是记录加的锁。

为了某事务的行为，INNODB添加了内部意向锁（表锁），在添加行锁之前添加。

意向共享锁（IS）：事务读、select

意向排他锁（IX）：事务写、insert、update、delete

间隙锁：用范围条件搜索是，键值在条件范围内，但不存在的记录加锁，会造成严重的锁等待。满足事务的隔离要求。因此尽量优化业务逻辑，使用相等条件。解决幻读场景。

其他锁

乐观锁：适用于多读场景，每次拿数据都认为别人没有更新数据。提高吞吐。

实现方式：1. 为数据增加一个版本标识，比如说添加一个version字段，每次更新加1；提交更新时，通过判断version是否一直进行比对然后更新；2. 使用时间戳，同理，更新时，检查时间戳尽心规避。

悲观锁：假定一定会发生并发冲突，传统关系型数据库都采用这种锁机制。
