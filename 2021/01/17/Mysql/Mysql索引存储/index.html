Mysql索引存储

Rrigyouu 2021-01-17 01:20:40

Mysql索引分类

索引分类 描述
单列索引 1. 普通索引：允许空值和重复值；
2. 唯一索引：值唯一，仅允许一个空值；
3. 主键索引：特殊唯一索引，不允许存在空值。
组合索引 多列索引，遵循最左前缀原则。
全文索引 MyISAM引擎，仅支持索引字段char、varchar、text，主要做关键字搜索
空间索引 空间数据类型的字段所做的索引

Mysql索引结构

哈希结构、B+树结构、R结构（空间索引，不常见）。

哈希结构

  1. 哈希表存储哈希值和链表指针；

首先通过key生成哈希值，然后找对应哈希值的链表指针；

若仅一行，则直接返回，若哈希冲突，则遍历key。

  1. 目前只有Memory引擎显示支持哈希索引，此外，Memory也支持B+树。

  2. 哈希结构优点：快，缺点：哈希索引不支持排序、匹配查找、只支持等值比较查询，因为数据不是顺序存储。

  3. 哈希结构使用场合：数据仓库应用中有一个经典的例子“星型”schema，需要关联很多表的查找，需要用到哈希索引。

B+树结构

  1. 平衡二叉树→B树→B+树。
树 结构
平衡二叉树 1. 节点最多允许两个子节点；
2. 左子<父亲<右子；
3. 左右子树高度相差不超过1，且子树也是平衡二叉树。
B树 1. 最多m个子节点；
2. 除根节点以外的非叶子节点最少有⌈m/2⌉个子节点（向下取整）；
3. 如果根节点不是叶子节点，则至少有两个子节点；
4. 拥有k个子节点的节点，有k-1个关键字；
5. 所有叶子结点都在同一层。
B+树 1. k 个子节点的非叶子节点拥有 k 个关键字，B树是k-1个；
2. 关键字不保存数据，只用来索引；
3. 数据保存在叶子结点。
4. 叶子结点有指针。

索引存储方式

存储方式 特性
非聚集索引 1. 早期的MYISAM引擎，且可以不用设主键；
2. 主键和二级索引一样，叶子结点存放地址。
聚集索引 1. 主流INNODB引擎，一张表只允许一个聚集索引；
2. 主键默认聚集索引，叶子结点存储主键key和其他行数据；
3. 二级索引存储索引key和主键key；
4. 如果通过主键搜索，则直接拿到数据，如果通过二级索引，则需要先找到主键key。

聚集索引优点：

  1. 相较与非聚集索引拿到的地址，还要进行一次IO，聚集索引明显更快；
  2. 非聚集索引在每次移动数据、数据页面分裂时，都需要维护二级索引，更改地址；但是InnoDB不需要更新二级索引，因为只保存了索引和主键值。
  3. 因为数据有序，主键的范围查询效率高；

聚集索引缺点：

  1. 插入的速度严重依赖于插入顺序，因为B+树需要主键进行排序；
  2. 当插入新行或者更新主键时，可能会导致“页分裂”问题，占用更多的磁盘空间；
  3. 全表扫描可能变慢，因为涉及到磁盘寻道；
  4. 二级索引保存了主键列，会占用更多的空间。

页

在mysql中有一个默认为1的参数innodb_file_per_table，该参数设定每个表当做单独的一个文件存储，可在/var/lib/mysql中查看。若有表database1：则有文件database1.ibd；dabase.frm。

database1 用途
database1.ibd Mysql数据文件和索引文件
database1.frm Mysql表结构文件

*.ibd由多个段segment组成，每个segment由区extent构成，每个区1MB，包含64个页，每个页占16KB。

页是InnoDB可操作的最小粒度。

页存在阈值属性merge_threshold，一般为50%。

页结构

前指针+【页目录分组（6个每组），用户数据】+后指针。

页就是B+树的叶子节点，页的查询可以通过新建索引文件来构建，这样就成了一个B+树。

页合并

插入数据，按照页的顺序，顺序插入；

删除数据时，数据并没有被删除，只是标记为删除，空间可被其他数据使用，当删除记录到达merge_threshold时，该页会找相邻页，凑一个整页，来进行页合并优化存储空间。

因此页合并发生在删除和更新操作中。

页分裂

插入数据，如果该页已满，下页又没有多余的存储空间，InnoDB会在这个页进行页分裂，创建新页，重新记录页之间的关系，造成页的错位。

页分裂发生在插入或者更新中。

分裂出来的页，想要恢复顺序只能页合并，或者OPTIMIZE重新整理表。

页合并和分裂过程中，需要在索引树上加锁，可能会导致索引的锁争用。如果更新操作没有页合并和分裂，成为乐观更新，只需要使用读锁（S）；反之为悲观更新，需要写锁（X）。
